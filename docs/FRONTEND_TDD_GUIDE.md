# FRONTEND_TDD_GUIDE

이 문서는 프론트엔드 구현을 TDD 방식으로 진행하기 위한 기준과 체크리스트입니다.
프레임워크와 테스트 도구는 최종 프론트 스택 결정 이후 확정합니다.

## 1. TDD 기본 규칙
- 기능 추가는 항상 테스트를 먼저 작성한다.
- 테스트는 실패 상태(RED) 확인 후 최소 구현(GREEN)으로 통과시킨다.
- 통과 후 리팩터링(REFACTOR) 단계에서 구조를 정리한다.

## 2. 테스트 레벨 정의
- 단위(Unit): 상태/로직/유틸 함수 검증
- 컴포넌트(Component): 사용자 상호작용과 UI 상태 변화 검증
- 통합(Integration): 라우팅/상태 전달/데이터 흐름 검증
- E2E: 핵심 사용자 시나리오 검증

## 3. 화면별 핵심 시나리오 테스트 (필수)

### Step 1. 서비스 모드 선택
- 모드 버튼 클릭 시 `comparisonMode`가 올바르게 설정됨
- COMPARE 선택 시 `/select-card`로 이동
- NEW 선택 시 `/categories`로 이동

### Step 2. 기존 카드 선택
- 검색 입력 시 리스트가 필터링됨
- 카드 선택 시 선택 상태 UI가 반영됨
- 카드 선택 전에는 '다음' 버튼이 비활성화됨
- 카드 선택 후에는 '다음' 버튼이 활성화됨

### Step 3. 혜택 카테고리 선택
- 2개 미만 선택 시 '다음' 비활성화
- 2~5개 선택 시 '다음' 활성화
- 6개 이상 선택 시 추가 선택이 차단되거나 오류 안내

### Step 4. 소비 금액 입력
- 슬라이더 변경 시 숫자 입력값이 동기화됨
- 숫자 입력 변경 시 슬라이더와 차트가 동기화됨
- 총합 금액이 실시간으로 갱신됨
- 토글 열림/닫힘 상태가 유지됨

### Step 5. 결과 화면
- NEW 모드: 3열 카드가 렌더링됨
- COMPARE 모드: 좌측 고정 카드 + 우측 캐러셀 구조가 렌더링됨
- '더 물어보기' 클릭 시 사이드 패널이 열림
- 패널 닫기 클릭 시 패널이 닫히고 배경 Dim이 해제됨

## 4. 전역 상태 전달 테스트 (필수)
- 입력 단계에서 만든 `selectedCategories`, `spendingData`가 결과 화면 계산 컨텍스트로 전달됨
- `selectedCurrentCard`는 COMPARE 모드에서만 요구됨

## 5. 테스트 우선순위
1. 전역 상태 전파 테스트
2. 라우팅 분기 테스트
3. 핵심 인터랙션 테스트
4. 비주얼 상태 변경 테스트

## 6. 구현 전 체크리스트
- 상태 스토어 구조 확정
- 라우팅 구조 확정
- 공용 컴포넌트 명세 확정
- 테스트 프레임워크 및 러너 확정
